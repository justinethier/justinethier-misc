Binding constructs

The binding constructs `=13`let, `=13`let*, `=13`letrec, `=13`letrec*, `=13`let-values, and `=13`let*-values give Scheme a block structure, like Algol 60. The syntax of the first four constructs is identical, but they differ in the regionsindexfile"region" "rm" aux they establish for their variable bindings. In a `=13`let expression, the initial values are computed before any of the variables become bound; in a `=13`let* expression, the bindings and evaluations are performed sequentially; while in `=13`letrec and `=13`letrec* expressions, all the bindings are in effect while their initial values are being computed, thus allowing mutually recursive definitions. The `=13`let-values and `=13`let*-values constructs are analogous to `=13`let and `=13`let* respectively, but are designed to handle multiple-valued expressions, binding different identifiers to the returned values.

Syntax: Bindings has the form

          gobblecr((variable init) ),
where each init is an expression, and body is a sequence of zero or more definitions followed by a sequence of one or more expressions as described in section 4.1.4. It is an error for a variable to appear more than once in the list of variables being bound.

Semantics: The inits are evaluated in the current environment (in some unspecified order), the variables are bound to fresh locations holding the results, the body is evaluated in the extended environment, and the values of the last expression of body are returned. (let ((x 2) (y 3))  (let ((x 7)        (z (+ x y)))    (* z x)))                   35

See also ``named `=13`let,'' section 4.2.4.






`=13`=0.5em(indexfile"let*" "tt" main let*bindings body) syntax

Syntax: Bindings has the form
unspecified error makeother^`=13` 


          gobblecr((variable init) ),
and body is a sequence of zero or more definitions followed by one or more expressions as described in section 4.1.4.

Semantics: The `=13`let* binding construct is similar to `=13`let, but the bindings are performed sequentially from left to right, and the regionindexfile"region" "rm" aux of a binding indicated by `=13`(variable init) is that part of the `=13`let* expression to the right of the binding. Thus the second binding is done in an environment in which the first binding is visible, and so on. The variables need not be distinct.


unspecified error makeother^`=13` 


          gobblecr(let ((x 2) (y 3))  (let* ((x 7)         (z (+ x y)))    (* z x)))             70






`=13`=0.5em(indexfile"letrec" "tt" main letrecbindings body) syntax

Syntax: Bindings has the form
unspecified error makeother^`=13` 


          gobblecr((variable init) ),
and body is a sequence of zero or more definitions followed by one or more expressions as described in section 4.1.4. It is an error for a variable to appear more than once in the list of variables being bound.

Semantics: The variables are bound to fresh locations holding unspecified values, the inits are evaluated in the resulting environment (in some unspecified order), each variable is assigned to the result of the corresponding init, the body is evaluated in the resulting environment, and the values of the last expression in body are returned. Each binding of a variable has the entire `=13`letrec expression as its regionindexfile"region" "rm" aux , making it possible to define mutually recursive procedures.


unspecified error makeother^`=13` 


          gobblecr(letrec ((even?          (lambda (n)            (if (zero? n)                #t                (odd? (- n 1)))))         (odd?          (lambda (n)            (if (zero? n)                #f                (even? (- n 1))))))  (even? 88))  #t

One restriction on `=13`letrec is very important: if it is not possible to evaluate each init without assigning or referring to the value of any variable, it is an error. The restriction is necessary because `=13`letrec is defined in terms of a procedure call where a `=13`lambda expression binds the variables to the values of the inits. In the most common uses of `=13`letrec, all the inits are `=13`lambda expressions and the restriction is satisfied automatically.






`=13`=0.5em(indexfile"letrec*" "tt" main letrec*bindings body) syntax

Syntax: Bindings has the form
unspecified error makeother^`=13` 


          gobblecr((variable init) ),
and bodyindexfile"body" "rm" aux is a sequence of zero or more definitions followed by one or more expressions as described in section 4.1.4. It is an error for a variable to appear more than once in the list of variables being bound.

Semantics: The variables are bound to fresh locations, each variable is assigned in left-to-right order to the result of evaluating the corresponding init, the body is evaluated in the resulting environment, and the values of the last expression in body are returned. Despite the left-to-right evaluation and assignment order, each binding of a variable has the entire `=13`letrec* expression as its regionindexfile"region" "rm" aux , making it possible to define mutually recursive procedures.

If it is not possible to evaluate each init without assigning or referring to the value of the corresponding variable or the variable of any of the bindings that follow it in bindings, it is an error. Another restriction is that it is an error to invoke the continuation of an init more than once.

          gobblecr(letrec* ((p           (lambda (x)             (+ 1 (q (- x 1)))))          (q           (lambda (y)             (if (zero? y)                 0                 (+ 1 (p (- y 1))))))          (x (p 5))          (y x))  y)                5





`=13`=0.5em(indexfile"let-values" "tt" main let-valuesmv binding spec body) syntax

Syntax: Mv binding spec has the form
unspecified error makeother^`=13` 


          gobblecr((formals init) ),

where each init is an expression, and body is zero or more definitions followed by a sequence of one or more expressions as described in section 4.1.4. It is an error for a variable to appear more than once in the set of formals.

Semantics: The inits are evaluated in the current environment (in some unspecified order) as if by invoking `=13`call-with-values, and the variables occurring in the formals are bound to fresh locations holding the values returned by the inits, where the formals are matched to the return values in the same way that the formals in a `=13`lambda expression are matched to the arguments in a procedure call. Then, the body is evaluated in the extended environment, and the values of the last expression of body are returned. Each binding of a variable has body as its region.indexfile"region" "rm" aux

It is an error if the formals do not match the number of values returned by the corresponding init.


unspecified error makeother^`=13` 


          gobblecr(let-values (((root rem) (exact-integer-sqrt 32)))  (* root rem))                35






`=13`=0.5em(indexfile"let*-values" "tt" main let*-valuesmv binding spec body) syntax

Syntax: Mv binding spec has the form
unspecified error makeother^`=13` 


          gobblecr((formals init) ),
and body is a sequence of zero or more definitions followed by one or more expressions as described in section 4.1.4. In each formals, it is an error if any variable appears more than once.

Semantics: The `=13`let*-values construct is similar to `=13`let-values, but the inits are evaluated and bindings created sequentially from left to right, with the region of the bindings of each formals including the inits to its right as well as body. Thus the second init is evaluated in an environment in which the first set of bindings is visible and initialized, and so on.


unspecified error makeother^`=13` 


          gobblecr(let ((a 'a) (b 'b) (x 'x) (y 'y))  (let*-values (((a b) (values x y))                ((x y) (values a b)))    (list a b x y)))     (x y x y)
