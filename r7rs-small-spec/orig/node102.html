Incompatibilities with RRS

This section enumerates the incompatibilities between RRS and the ``Revised report'' [33] and its accompanying Standard Libraries document.

This list is not authoritative, and is possibly incomplete.

  • RRS libraries begin with the keyword `=13`define-library rather than `=13`library in order to make them syntactically distinguishable from RRS libraries. In RRS terms, the body of an RRS library consists of a single export declaration followed by a single import declaration, followed by commands and definitions. In RRS, commands and definitions are not permitted directly within the body: they have to be be wrapped in a `=13`begin library declaration.

  • There is no direct RRS equivalent of the `=13`include, `=13`include-ci, `=13`include-library-declarations, or `=13`cond-expand library declarations. On the other hand, the RRS library syntax does not support phase or version specifications.

  • The grouping of standardized identifiers into libraries is different from the RRS approach. In particular, procedures which are optional in RRS either expressly or by implication, have been removed from the base library. Only the base library itself is an absolute requirement.

  • No form of identifier syntax is provided.

  • Internal syntax definitions are allowed, but uses of a syntax form cannot appear before its definition; the `=13`even/`=13`odd example given in RRS is not allowed.

  • The RRS exception system was incorporated as-is, but the condition types have been left unspecified. In particular, where RRS requires a condition of a specified type to be signaled, RRS says only ``it is an error'', leaving the question of signaling open.

  • Full Unicode support is not required. Normalization is not provided. Character comparisons are defined by Unicode, but string comparisons are implementation-dependent. Non-Unicode characters are permitted.

  • The full numeric tower is optional as in RRS, but optional support for IEEE infinities, NaN, and -0.0 was adopted from RRS. Most clarifications on numeric results were also adopted, but the RRS procedures `=13`real-valued?, `=13`rational-valued?, and `=13`integer-valued? were not. The RRS division operators `=13`div, `=13`mod, `=13`div-and-mod, `=13`div0, `=13`mod0 and `=13`div0-and-mod0 are not provided.

  • When a result is unspecified, it is still required to be a single value. However, non-final expressions in a body can return any number of values.

  • The semantics of `=13`map and `=13`for-each have been changed to use the SRFI 1 [30] early termination behavior. Likewise, `=13`assoc and `=13`member take an optional `=13`equal? argument as in SRFI 1, instead of the separate `=13`assp and `=13`memp procedures of RRS.

  • The RRS `=13`quasiquote clarifications have been adopted, with the exception of multiple-argument `=13`unquote and `=13`unquote-splicing.

  • The RRS method of specifying mantissa widths was not adopted.

  • String ports are compatible with SRFI 6 [7] rather than RRS.

  • RRS-style bytevectors are included, but only the unsigned byte (`=13`u8) procedures have been provided. The lexical syntax uses `=13`#u8 for compatibility with SRFI 4 [13], rather than the RRS `=13`#vu8 style.

  • The utility macros `=13`when and `=13`unless are provided, but their result is left unspecified.

  • The remaining features of the Standard Libraries document were left to future standardization efforts.
