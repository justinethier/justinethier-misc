Incompatibilities with RRS

This section enumerates the incompatibilities between this report and the ``Revised report'' [20].

This list is not authoritative, but is believed to be correct and complete.

  • Case sensitivity is now the default in symbols and character names. This means that code written under the assumption that symbols could be written `=13`FOO or `=13`Foo in some contexts and `=13`foo in other contexts can either be changed, be marked with the new `=13`#!fold-case directive, or be included in a library using the `=13`include-ci library declaration. All standard identifiers are entirely in lower case.

  • The `=13`syntax-rules construct now recognizes _ (underscore) as a wildcard, which means it cannot be used as a syntax variable. It can still be used as a literal.

  • The RRS procedures `=13`exact->inexact and `=13`inexact->exact have been renamed to their RRS names, `=13`inexact and `=13`exact, respectively, as these names are shorter and more correct. The former names are still available in the RRS library.

  • The guarantee that string comparison (with `=13`string<? and the related predicates) is a lexicographical extension of character comparison (with `=13`char<? and the related predicates) has been removed.

  • Support for the # character in numeric literals is no longer required.

  • Support for the letters `=13`s, `=13`f, `=13`d, and `=13`l as exponent markers is no longer required.

  • Implementations of `=13`string->->number are no longer permitted to return #f when the argument contains an explicit radix prefix, and must be compatible with `=13`read and the syntax of numbers in programs.

  • The procedures `=13`transcript-on and `=13`transcript-off have been removed.
