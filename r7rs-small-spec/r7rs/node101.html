Other language changes since RRS

This section enumerates the additional differences between this report and the ``Revised report'' [20].

This list is not authoritative, but is believed to be correct and complete.

  • Various minor ambiguities and unclarities in RRS have been cleaned up.

  • Libraries have been added as a new program structure to improve encapsulation and sharing of code. Some existing and new identifiers have been factored out into separate libraries. Libraries can be imported into other libraries or main programs, with controlled exposure and renaming of identifiers. The contents of a library can be made conditional on the features of the implementation on which it is to be used. There is an RRS compatibility library.

  • The expressions types `=13`include, `=13`include-ci, and `=13`cond-expand have been added to the base library; they have the same semantics as the corresponding library declarations.

  • Exceptions can now be signaled explicitly with `=13`raise, `=13`raise-continuable or `=13`error, and can be handled with `=13`with-exception-handler and the `=13`guard syntax. Any object can specify an error condition; the implementation-defined conditions signaled by `=13`error have a predicate to detect them and accessor functions to retrieve the arguments passed to `=13`error. Conditions signaled by `=13`read and by file-related procedures also have predicates to detect them.

  • New disjoint types supporting access to multiple fields can be generated with the `=13`define-record-type of SRFI 9 [19]

  • Parameter objects can be created with `=13`make-parameter, and dynamically rebound with `=13`parameterize. The procedures `=13`current-input-port and `=13`current-output-port are now parameter objects, as is the newly introduced `=13`current-error-port.

  • Support for promises has been enhanced based on SRFI 45 [38].

  • Bytevectors, vectors of exact integers in the range from 0 to 255 inclusive, have been added as a new disjoint type. A subset of the vector procedures is provided. Bytevectors can be converted to and from strings in accordance with the UTF-8 character encoding. Bytevectors have a datum representation and evaluate to themselves.

  • Vector constants evaluate to themselves.

  • The procedure `=13`read-line is provided to make line-oriented textual input simpler.

  • The procedure `=13`flush-output-port is provided to allow minimal control of output port buffering.

  • Ports can now be designated as textual or binary ports, with new procedures for reading and writing binary data. The new predicates `=13`input-port-open? and `=13`output-port-open? return whether a port is open or closed. The new procedure `=13`close-port now closes a port; if the port has both input and output sides, both are closed.

  • String ports have been added as a way to read and write characters to and from strings, and bytevector ports to read and write bytes to and from bytevectors.

  • There are now I/O procedures specific to strings and bytevectors.

  • The `=13`write procedure now generates datum labels when applied to circular objects. The new procedure `=13`write-simple never generates labels; `=13`write-shared generates labels for all shared and circular structure. The `=13`display procedure must not loop on circular objects.

  • The RRS procedure `=13`eof-object has been added. Eof-objects are now required to be a disjoint type.

  • Syntax definitions are now allowed wherever variable definitions are.

  • The `=13`syntax-rules construct now allows the ellipsis symbol to be specified explicitly instead of the default `=13`..., allows template escapes with an ellipsis-prefixed list, and allows tail patterns to follow an ellipsis pattern.

  • The `=13`syntax-error syntax has been added as a way to signal immediate and more informative errors when a macro is expanded.

  • The `=13`letrec* binding construct has been added, and internal `=13`define is specified in terms of it.

  • Support for capturing multiple values has been enhanced with `=13`define-values, `=13`let-values, and `=13`let*-values. Standard expression types which contain a sequence of expressions now permit passing zero or more than one value to the continuations of all non-final expressions of the sequence.

  • The `=13`case conditional now supports => syntax analogous to `=13`cond not only in regular clauses but in the `=13`else clause as well.

  • To support dispatching on the number of arguments passed to a procedure, `=13`case-lambda has been added in its own library.

  • The convenience conditionals `=13`when and `=13`unless have been added.

  • The behavior of `=13`eqv? on inexact numbers now conforms to the RRS definition.

  • When applied to procedures, `=13`eq? and `=13`eqv? are permitted to return different answers.

  • The RRS procedures `=13`boolean=? and `=13`symbol=? have been added.

  • Positive infinity, negative infinity, NaN, and negative inexact zero have been added to the numeric tower as inexact values with the written representations +inf.0, -inf.0, +nan.0, and `=13`-0.0 respectively. Support for them is not required. The representation -nan.0 is synonymous with +nan.0.

  • The `=13`log procedure now accepts a second argument specifying the logarithm base.

  • The procedures `=13`map and `=13`for-each are now required to terminate on the shortest argument list.

  • The procedures `=13`member and `=13`assoc now take an optional third argument specifying the equality predicate to be used.

  • The numeric procedures `=13`finite?, `=13`infinite?, `=13`nan?, `=13`exact-integer?, `=13`square, and `=13`exact-integer-sqrt have been added.

  • The `=13`- and `=13`/ procedures and the character and string comparison predicates are now required to support more than two arguments.

  • The forms #true and #false are now supported as well as #t and #f.

  • The procedures `=13`make-list, `=13`list-copy, `=13`list-set!, `=13`string-map, `=13`string-for-each, `=13`string->vector, `=13`vector-append, `=13`vector-copy, `=13`vector-map, `=13`vector-for-each, `=13`vector->string, `=13`vector-copy!, and `=13`string-copy! have been added to round out the sequence operations.

  • Some string and vector procedures support processing of part of a string or vector using optional start and end arguments.

  • Some list procedures are now defined on circular lists.

  • Implementations may provide any subset of the full Unicode repertoire that includes ASCII, but implementations must support any such subset in a way consistent with Unicode. Various character and string procedures have been extended accordingly, and case conversion procedures added for strings. String comparison is no longer required to be consistent with character comparison, which is based solely on Unicode scalar values. The new `=13`digit-value procedure has been added to obtain the numerical value of a numeric character.

  • There are now two additional comment syntaxes: #; to skip the next datum, and #| ... |# for nestable block comments.

  • Data prefixed with datum labels #<n>= can be referenced with #<n>#, allowing for reading and writing of data with shared structure.

  • Strings and symbols now allow mnemonic and numeric escape sequences, and the list of named characters has been extended.

  • The procedures `=13`file-exists? and `=13`delete-file are available in the (scheme file) library.

  • An interface to the system environment, command line, and process exit status is available in the (scheme process-context) library.

  • Procedures for accessing time-related values are available in the (scheme time) library.

  • A less irregular set of integer division operators is provided with new and clearer names.

  • The `=13`load procedure now accepts a second argument specifying the environment to load into.

  • The `=13`call-with-current-continuation procedure now has the synonym `=13`call/cc.

  • The semantics of read-eval-print loops are now partly prescribed, requiring the redefinition of procedures, but not syntax keywords, to have retroactive effect.

  • The formal semantics now handles `=13`dynamic-wind.
