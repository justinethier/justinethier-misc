Library Syntax

A library definition takes the following form: indexfile"define-library" "tt" main


unspecified error makeother^`=13` 


          gobblecr(define-library library name  library declaration )

library name is a list whose members are identifiers and exact non-negative integers. It is used to identify the library uniquely when importing from other programs or libraries. Libraries whose first identifier is `=13`scheme are reserved for use by this report and future versions of this report. Libraries whose first identifier is `=13`srfi are reserved for libraries implementing Scheme Requests for Implementation. It is inadvisable, but not an error, for identifiers in library names to contain any of the characters `=13`| \ ? * < " : > + [ ] / or control characters after escapes are expanded.

A library declaration is any of:

  • (export export spec )

  • (import import set )

  • (begin command or definition )

  • (include filename filename )

  • (include-ci filename filename )

  • (include-library-declarations filename filename )

  • (cond-expand ce-clause ce-clause )

An indexfile"export" "tt" aux export declaration specifies a list of identifiers which can be made visible to other libraries or programs. An export spec takes one of the following forms:

  • identifier
  • (rename identifier identifier)

In an export spec, an identifier names a single binding defined within or imported into the library, where the external name for the export is the same as the name of the binding within the library. A indexfile"rename" "tt" aux rename spec exports the binding defined within or imported into the library and named by identifier in each (identifier identifier) pairing, using identifier as the external name.

An indexfile"import" "tt" aux import declaration provides a way to import the identifiers exported by another library. It has the same syntax and semantics as an import declaration used in a program or at the REPL (see section 5.2).

The indexfile"begin" "tt" aux begin, indexfile"include" "tt" aux include, and indexfile"include-ci" "tt" aux include-ci declarations are used to specify the body of the library. They have the same syntax and semantics as the corresponding expression types. This form of `=13`begin is analogous to, but not the same as, the two types of `=13`begin defined in section 4.2.3.

The indexfile"include-library-declarations" "tt" aux include-library-declarations declaration is similar to indexfile"include" "tt" aux include except that the contents of the file are spliced directly into the current library definition. This can be used, for example, to share the same indexfile"export" "tt" aux export declaration among multiple libraries as a simple form of library interface.

The indexfile"cond-expand" "tt" aux cond-expand declaration has the same syntax and semantics as the indexfile"cond-expand" "tt" aux cond-expand expression type, except that it expands to spliced-in library declarations rather than expressions enclosed in `=13`begin.

One possible implementation of libraries is as follows: After all indexfile"cond-expand" "tt" aux cond-expand library declarations are expanded, a new environment is constructed for the library consisting of all imported bindings. The expressions from all indexfile"begin" "tt" aux begin, indexfile"include" "tt" aux include and indexfile"include-ci" "tt" aux include-ci library declarations are expanded in that environment in the order in which they occur in the library. Alternatively, indexfile"cond-expand" "tt" aux cond-expand and indexfile"import" "tt" aux import declarations may be processed in left to right order interspersed with the processing of other declarations, with the environment growing as imported bindings are added to it by each indexfile"import" "tt" aux import declaration.

When a library is loaded, its expressions are executed in textual order. If a library's definitions are referenced in the expanded form of a program or library body, then that library must be loaded before the expanded program or library body is evaluated. This rule applies transitively. If a library is imported by more than one program or library, it may possibly be loaded additional times.

Similarly, during the expansion of a library `=13`(foo), if any syntax keywords imported from another library `=13`(bar) are needed to expand the library, then the library `=13`(bar) must be expanded and its syntax definitions evaluated before the expansion of `=13`(foo).

Regardless of the number of times that a library is loaded, each program or library that imports bindings from a library must do so from a single loading of that library, regardless of the number of import declarations in which it appears. That is, `=13`(import (only (foo) a)) followed by `=13`(import (only (foo) b)) has the same effect as `=13`(import (only (foo) a b)).
